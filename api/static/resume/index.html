02 / Design Rationale

What, How & Why

01

What

  • Short link creation with collision-safe shortcode generation
  • Redirect via GET /{shortcode} with active/expiry checks
  • Stats and management via /links/{shortcode}/stats
02

How

  • Cache-aside reads in Redis for low-latency redirects
  • Click events published to RabbitMQ without blocking the response
  • Worker batches click updates and flushes to Postgres periodically
03

Why

  • Keeps the hot redirect path fast under load
  • Isolates analytics from user-facing latency for resilience
  • Full observability via traces, metrics, and structured logs

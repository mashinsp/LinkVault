02 / What / How / Why

Design Summary

What

  • Create short links with collision-safe shortcode generation.
  • Redirect via GET /{shortcode} with active/expiry checks.
  • Expose stats and management via /links/{shortcode}/stats and delete endpoints.

How

  • Cache-aside reads in Redis for low-latency redirects.
  • Click events published to RabbitMQ without blocking redirect response.
  • Worker batches click updates and flushes to Postgres every few seconds.

Why

  • Keeps the hot redirect path fast under load.
  • Improves resilience by isolating analytics from user-facing latency.
  • Enables production debugging with traces, metrics, and structured logs.
